# SysMLv2モデル生成ワークフロー

この資料は、`report/` 配下の方法論を、実際のモデル生成で使いやすい形に要約したものである。

## 1. 入力の扱い

- 入力は原則として**プロンプト本文**または**指定ファイルのテキスト内容**とする。
- ファイル入力は、Markdown / テキスト / 既存 `.sysml` を優先する。
- 長文は一括投入せず、見出し単位または意味ブロック単位で分割する。
- 原文のモダリティ（例: 「しなければならない」）は保持する。

## 2. 変換時の基本姿勢

- LLM は支援手段であり、意味の確定責任は人間が負う。
- 出力は完成品ではなく、**レビュー可能な提案**として扱う。
- 不明点は埋めずに `<T.B.D>` や要確認として残す。

## 3. A/B/C 三層の整理

### A 層

- 要求
- 制約
- ドメイン概念
- 受入条件

### B 層

- 設計判断
- 責務配賦
- 採用理由 / 不採用理由
- トレードオフ
- 検証戦略

### C 層

- SysMLv2 要素
- 構造
- インターフェース
- 振る舞い
- 状態
- テスト構成

## 4. PIM/PSM の分け方

### PIM

- 論理責務
- 論理構造
- 論理インターフェース
- 意味的状態 / 遷移
- 論理制約

### PSM

- 実行構造
- 実行インターフェース
- 実行振る舞い
- 性能 / タイミング / 資源 / 運用制約

## 5. 生成時の優先順位

1. 要求と制約を固定する
2. 判断理由を抽出する
3. 構造と振る舞いへ写像する
4. 要求↔判断↔構造の対応を明示する
5. 未確定事項とレビュー項目を残す

## 6. 推奨 SysMLv2 要素

- `package`
- `requirement def`, `requirement`
- `constraint def`
- `part def`, `part`
- `item def`
- `interface def`
- `port def`
- `action def`
- `state def`
- `satisfy`, `refine`, `allocate`, `dependency`, `verify`

## 7. 出力時の最小セット

小さな入力でも、最低限次を返す。

- 生成対象の要約
- 前提 / 仮定
- SysMLv2 コード
- 要レビュー点

## 8. 長文入力の扱い

長文または複数章入力では、次の順で行う。

1. ブロック分割
2. 各ブロックの A/B/C 整理
3. ブロックごとの SysMLv2 候補生成
4. 命名と用語の統合
5. パッケージ統合
6. トレース見直し

## 9. 避けるべきこと

- 根拠のない要素追加
- 実装依存の早すぎる混入
- B 層なしの C 層直生成
- 用語の揺れ放置
- レビュー前提を隠した断定的出力
